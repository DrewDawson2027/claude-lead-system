# Architecture Comparison: Native Agent Teams vs. Lead System

_Technical reference for onboarding and implementation decisions._
_Last updated: 2026-03-15_

This document is a point-in-time architecture comparison, not a parity scorecard or replacement claim.

---

## 1. Architecture Overview

### Native Agent Teams

```
┌─────────────────────────────────────────────────────────┐
│  Lead Process (Node.js / React + Ink)                   │
│                                                         │
│  selectedTeammate: React state (in-process switch)      │
│                                                         │
│  child_process.spawn() ──► teammate-a  (Node.js child)  │
│                       ──► teammate-b  (Node.js child)   │
│                       ──► teammate-c  (Node.js child)   │
│                                                         │
│  Inbox: ~/.claude/teams/{name}/{agent-id}/inbox.json    │
│         (JSON array, full read/write per message)       │
│                                                         │
│  Display:                                               │
│    in-process  → React state swap (selectedTeammate)    │
│    split-pane  → separate tmux panes                    │
└─────────────────────────────────────────────────────────┘
```

Key properties:

- All teammates run as Node.js child processes inside the same Claude Code session.
- The lead switches visible context by updating `selectedTeammate` React state; there is no separation of address spaces.
- Inboxes are JSON arrays stored at `~/.claude/teams/{name}/{agent-id}/inbox.json`. Every message requires reading the full array, appending, and writing the whole file back (O(N) per message).
- Resume state is stored per-agent; context dropping on resume is a known bug (see §3).

### Lead System

```
┌─────────────────────────────────────────────────────────┐
│  Lead Process (Claude Code + MCP coordinator)           │
│                                                         │
│  MCP tools: coord_spawn_worker, coord_send_message, …   │
│                                                         │
│  coord_spawn_worker ──► worker-a (claude -p, stateless) │
│                    ──► worker-b (claude -p, stateless)  │
│                    ──► worker-c (interactive + script)  │
│                                                         │
│  Inbox: ~/.claude/terminals/inbox/{session-id}.jsonl    │
│         (JSONL, O(1) append per message)                │
│                                                         │
│  Display:                                               │
│    in-process  → tmux capture-pane (read-only poll)     │
│    split-pane  → tmux split-window + select-layout tiled│
└─────────────────────────────────────────────────────────┘
```

Key properties:

- Workers are independent processes spawned by shell script; the lead has no in-process relationship with them.
- Workers run `claude -p` (pipe / stateless) or interactive mode with `script` transcript capture.
- Inboxes are JSONL files; each message is a single `appendFileSync` (O(1)).
- Session IDs are tracked per-worker (`--session-id`); true resume via `--resume <session-id>` is supported.
- Coordination metadata on the filesystem path (task status, inbox delivery, plan approval) consumes **0 model/API tokens**.

## 3. Known Native Bugs We've Fixed

### #27265 / #29548 — Plan auto-approve (plan mode skipped)

**Native behavior:** When a teammate runs in plan mode, the approval gate can be bypassed silently; the teammate proceeds to edits without waiting for lead confirmation.

**Our fix:** Workers running with `require_plan: true` write their plan to `~/.claude/terminals/results/{task-id}.plan.md`, then send `[PLAN READY]` to the lead inbox and hard-wait for an inbox message containing `[APPROVED]` or `[REVISION]`. The approval file (`{task-id}.approval`) is also written by `coord_approve_plan` as a secondary check. No file edits begin until the explicit signal is received.

---

### #25135 — SendMessage silent failure

**Native behavior:** `SendMessage` with an unknown `agentId` or `teamName` returns success without delivering the message; the sender has no indication of failure.

**Our fix:** `coord_send_message` runs `resolveWorkerName(targetName)` before writing to the inbox. If the worker name cannot be resolved to an active session (checked against session files and meta files), the call returns:

```
Worker name "<name>" not found. Use coord_list_sessions to find active workers.
```

No message is written. The lead knows immediately.

---

### #15837 / #10856 — Resume drops context

**Native behavior:** Resuming a teammate after a crash or restart can drop prior conversation context depending on how the agent state was persisted.

**Our fix:** Interactive workers are spawned with `--session-id <uuid>` (stored in `meta.claude_session_id`). `coord_resume_worker` checks for a stored session ID and invokes `claude --resume <session-id>` which reloads the full conversation from Claude's session store. If no session ID is available (e.g., pipe-mode workers), it falls back to a continuation prompt that includes the full transcript (`{task-id}.transcript`, up to 30 KB) captured by `script`.

---

### Inbox O(N) write amplification

**Native behavior:** The native inbox is a JSON array. Reading, appending, and writing the array back scales linearly with inbox depth. Under high message volume (e.g., broadcast to all teammates), the last recipient pays the full cost of all prior messages.

**Our fix:** Inboxes are JSONL files. Each message is a single `appendJSONLineSecure(inboxFile, msg)` call — one `appendFileSync`. Reading is handled by `readJSONLLimited` which enforces a per-read line cap to prevent runaway memory use. Drain (`coord_check_inbox`) atomically renames the file before reading, so concurrent writers don't lose messages.

## 4. Spawn Command Reference

### Native spawn command (from bug #24989)

```sh
claude \
  --agent-id qa@myteam \
  --agent-name qa \
  --team-name myteam \
  --agent-color purple \
  --parent-session-id <uuid> \
  --agent-type general-purpose \
  --dangerously-skip-permissions \
  --model claude-opus-4-6
```

Properties:

- Runs as a managed child process inside the lead's Node.js runtime.
- Agent ID, team membership, and color are registered with the native runtime at spawn time.
- `--dangerously-skip-permissions` bypasses all permission prompts for the teammate.
- Context is inherited from the parent session; `--parent-session-id` ties the child to the lead's conversation.

---

### Our spawn command (pipe / stateless mode)

From `buildWorkerScript` in `lib/platform/common.js`:

```sh
cd <workDir> \
  && echo "Worker <taskId> starting at $(date)" > <resultFile> \
  && echo $$ > <pidFile> \
  && export CLAUDE_AUTOCLAIM_NODE=... \
  && export CLAUDE_AUTOCLAIM_SCRIPT=... \
  && export CLAUDE_AUTOCLAIM_ARGS_B64=<base64-payload> \
  && export CLAUDE_WORKER_TASK_ID=<taskId> \
  && export CLAUDE_WORKER_NAME=<workerName> \
  && export CLAUDE_LEAD_SESSION_ID=<leadSessionId> \
  && export CLAUDE_LEAD_PANE_ID=<leadPaneId> \
  && unset CLAUDECODE \
  && claude -p \
       --model <model> \
       [--agent <agent>] \
       [--settings <settingsFile>] \
       < <promptFile> \
       >> <resultFile> 2>&1 \
  && printf '{"status":"completed","finished":"%s","task_id":"%s"}' \
       "$(date -u +%Y-%m-%dT%H:%M:%SZ)" <taskId> > <metaFile>.done \
  && rm -f <pidFile> \
  && ([ -n "${CLAUDE_AUTOCLAIM_ARGS_B64:-}" ] \
       && "$CLAUDE_AUTOCLAIM_NODE" "$CLAUDE_AUTOCLAIM_SCRIPT" >/dev/null 2>&1) || true
```

### Our spawn command (interactive mode)

From `buildInteractiveWorkerScript` in `lib/platform/common.js`:

```sh
cd <workDir> \
  && echo $$ > <pidFile> \
  && export CLAUDE_WORKER_TASK_ID=<taskId> \
  && export CLAUDE_WORKER_NAME=<workerName> \
  && export CLAUDE_LEAD_SESSION_ID=<leadSessionId> \
  && export CLAUDE_LEAD_PANE_ID=<leadPaneId> \
  && [idle-detector subprocess in background] \
  && trap '[completion + autoclaim]' EXIT \
  && while true; do
       WORKER_PROMPT=$(cat <promptFile>)
       unset CLAUDECODE
       script -q <transcriptFile> \
         claude --prompt "$WORKER_PROMPT" \
                --permission-mode <mode> \
                --model <model> \
                --session-id <uuid> \
                [--agent <agent>]
       [write .meta.json.done]
       [rm pidFile]
       [append [COMPLETED] to lead inbox]
       [claim-next-task block]
     done
```

### Our resume command

From `buildResumeWorkerScript` in `lib/platform/common.js`:

```sh
cd <workDir> \
  && echo $$ > <pidFile> \
  && [env exports] \
  && trap '[completion + autoclaim]' EXIT \
  && [idle-detector subprocess] \
  && unset CLAUDECODE \
  && claude --resume <session-id> [--settings <settingsFile>]
```

Properties common to all our modes:

- `unset CLAUDECODE` prevents the child from inheriting the parent session's environment variables, which would cause the child to appear as a continuation of the lead's own session.
- `CLAUDE_AUTOCLAIM_ARGS_B64` carries the base64-encoded autoclaim payload; after each task completes, `claim-next-task.mjs` claims the next pending task from the board and re-uses the running process rather than spawning a new one.
- PID is written to `{taskId}.pid` immediately after `cd`; liveness is checked by `kill -0 <pid>`.
- Completion signals are written to `{taskId}.meta.json.done` (not the meta file itself) to allow atomic detection by the lead.
- `script` (macOS/Linux) captures the full terminal transcript to `{taskId}.transcript` for resume support; on Linux the `-c` flag variant is used.

## 5. File Layout Reference

### Native Agent Teams

```
~/.claude/teams/
  {team-name}/
    {agent-id}/
      inbox.json          # JSON array of messages (O(N) read/write)
      state.json          # agent state snapshot
```

### Lead System

```
~/.claude/terminals/
  session-{id}.json       # session heartbeat / status
  inbox/
    {session-id}.jsonl    # per-session JSONL inbox (O(1) append)
  tasks/
    {task-id}.json        # task board entries
  teams/
    {team-name}.json      # team config + policy + member roster
  results/
    {task-id}.txt         # stdout/stderr capture
    {task-id}.meta.json   # spawn metadata (model, mode, files, …)
    {task-id}.meta.json.done  # completion sentinel (written atomically)
    {task-id}.pid         # worker PID (deleted on exit)
    {task-id}.prompt      # full prompt written to disk before spawn
    {task-id}.transcript  # script(1) full terminal transcript
    {task-id}.approval    # plan approval status (approved / revision_requested)
    {task-id}.audit.jsonl # per-task audit trail
    {task-id}.handoff.json # reassignment snapshot
  context/
    lead-context-{session-id}.json  # lead's exported context for workers
    {team-name}.json                # shared team context store
  reports/
    {task-id}.jsonl       # worker progress reports (write from worker, read by lead)
```
